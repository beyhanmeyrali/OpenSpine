# v0.1 — Foundation

**Status:** Planning
**Target window:** Q3 2026
**Theme:** "First runnable OpenSpine."

This is the milestone that turns OpenSpine from a spec into a thing you can `docker compose up` and post against. It carries no business logic of consequence — its job is to **prove the spine works**: identity → tenancy → master-data core → dual-write to Qdrant → plugin host alive → one example plugin loaded → audit log capturing.

Once v0.1 ships, every later milestone is incremental. The hardest architectural questions are answered here.

---

## 1. What v0.1 delivers

### 1.1 Runnable system

- `docker compose up` brings up Postgres 16, Qdrant, Redis Streams, the FastAPI app, and an embedding worker.
- A web UI doesn't exist yet — v0.1 is API-only. (The React frontend lands with v0.2 once there are user-visible business transactions to render.)
- A health endpoint, an OpenAPI doc at `/docs`, and a Prometheus `/metrics` endpoint.

### 1.2 Identity & tenancy

- `id_tenant`, `id_principal` (human + agent + technical), `id_credential`, `id_session`, `id_token`, `id_audit_event`.
- Auth methods in v0.1: **password+TOTP** for humans, **API tokens** for technical accounts, **agent tokens** with scoped permissions for agents. SSO/passkey defer to v0.2.
- Multi-tenant by default. Three-layer isolation (RLS + service filter + Qdrant tenant scoping) wired from day one — adding tenants later is a one-row insert, not a migration.
- A minimal RBAC: `id_role_single`, `id_role_composite`, `id_principal_role`, `id_auth_object`. Full Phase-1 role catalogue is **seeded** but only the system-administration single roles are exercised in v0.1.

### 1.3 Master Data — minimum cut

Just enough to make v0.2 (Finance core) implementable:

- `md_company_code` — legal entity with currency + CoA + fiscal-year-variant assignment
- `md_chart_of_accounts`, `md_gl_account`, `md_gl_account_company`, `md_account_group`
- `md_business_partner` + `md_bp_role` + `md_bp_address` + `md_bp_bank` (vendor / customer / employee roles)
- `md_material` + `md_material_plant` + `md_material_valuation` + `md_material_uom` (with one global UoM catalogue: `md_uom`, `md_uom_conversion`)
- `md_currency`, `md_exchange_rate_type`, `md_fx_rate`
- `md_number_range`, `md_fiscal_year_variant`, `md_posting_period`
- `md_plant`, `md_storage_location`, `md_purchasing_org`, `md_purchasing_group`, `md_controlling_area`, `md_factory_calendar`

Master-data **transactions** that work end-to-end in v0.1: create tenant, create Company Code, create CoA + GL accounts, create BP (vendor + customer roles), create Material (basic + plant + valuation views), upload FX rates, open first posting period.

Deferred from MD until v0.2+: BP merge/split, material flag-for-deletion lifecycle, complex number-range scoping decisions, address normalisation.

### 1.4 Dual-write pipeline operational

- Every business write to Postgres publishes a `master_data.<entity>.created|updated` event on Redis Streams.
- The embedding worker consumes events, calls Qwen3-Embedding via Ollama, upserts into a **per-tenant Qdrant collection** with the entity's payload + custom-field extensions.
- Reconciliation job (hourly, manually triggerable) replays events for any entity not yet in Qdrant.
- "Lose Qdrant, run reconciliation, system is whole again" is a **tested** scenario before v0.1 ships.

### 1.5 Plugin host

- Discovery via Python entry points (`openspine.plugins`).
- `plugin.yaml` parsing with strict compatibility-range enforcement.
- Hook dispatcher: `pre_*` hooks run inline in the service transaction; `post_*` hooks dispatched via Redis Streams to async handlers.
- Custom-field surface: `extend_entity()` adds `ext_<plugin>_<field>` columns via Alembic-managed migrations the plugin owns.
- One example plugin ships in `examples/openspine-plugin-example/` exercising every extension mechanism: a hook subscription, a custom field, a custom endpoint, a custom auth object. This is the **reference** for plugin authors.

### 1.6 Agent surface — minimum viable

Per the AI-first non-negotiable, agents must be a first-class citizen from v0.1, not bolted on at v0.5. The minimum:

- Agent tokens issued through the same API that creates API tokens, with the additional scope/expiry/reasoning fields.
- Every API response includes a `_meta` block describing available next actions and field semantics — the **self-describing** contract.
- Every error is structured (`error`, `domain`, `action`, `reason`, `attempted`, `allowed`, `principal_id`, `trace_id`).
- `id_agent_decision_trace` table exists; agents can write traces linked to `id_audit_event` rows.
- Hybrid search exists for master data: `GET /md/search?q=...` returns Qdrant candidates plus the verifying Postgres rows.

Deferred to v0.5: NL reporting, agentic document understanding (invoice from PDF), domain-specific agent operators.

### 1.7 Observability

- OpenTelemetry traces on every request, propagated through the event bus to the embedding worker.
- Structured JSON logs.
- Prometheus metrics: request rate / latency / error per endpoint, event-bus consumer lag, Qdrant index lag, hook execution time per hook.

### 1.8 Documentation

- `docs/architecture/data-model.md` written — the now-missing-but-referenced doc.
- ADRs for the four already-implicit decisions land alongside v0.1 (`0001-postgres-as-source-of-truth`, `0002-qdrant-over-pgvector`, `0003-universal-journal`, `0004-agpl-license` — last one resolves the private-plugin question).
- `docs/architecture/event-catalogue.md` written for the events v0.1 emits.
- `docs/architecture/plugin-system.md` written, with the example plugin as the running illustration.

## 2. Out of scope for v0.1

Stated explicitly so scope creep has a clear boundary:

- Any FI posting (no journal entries, no `fin_document_*` rows). FI is v0.2.
- Any CO master data or allocations. v0.2.x.
- Any MM transaction (no PR/PO/GR/IR). v0.3.
- Any PP transaction (no BOM/MRP/production order). v0.4.
- React frontend. Lands with v0.2 alongside the first user-visible business transactions.
- SSO / OIDC / SAML / passkey. v0.2.
- Bank accounting, EBS import. v0.5+.
- Asset accounting. Post-v0.4.

## 3. Definition of done

v0.1 is done when **all** of these are true:

1. `docker compose up` brings the system up on a clean machine in < 5 minutes.
2. The end-to-end happy path runs from a single integration test:
   - Create tenant → create Company Code → create CoA + 3 GL accounts → create vendor BP → create material → upload FX rates → open posting period.
   - Every entity is queryable via REST.
   - Every entity is searchable via the hybrid `/md/search` endpoint within 10 seconds of creation.
3. The example plugin loads, registers its hook, fires it on `business_partner.pre_save`, adds its custom field, exposes its custom endpoint.
4. Killing Qdrant and running reconciliation rebuilds the index without manual intervention.
5. `id_audit_event` records every authentication, every authorisation decision, every business-data write.
6. RLS prevents cross-tenant reads in a test that explicitly tries.
7. SoD baseline is loaded; a test that grants `MD_BP_CREATE` + `FI_AP_PAYMENT_RELEASE` to the same principal+scope is **blocked**.
8. OpenAPI doc renders. Prometheus metrics scrape. OTel traces span request → service → DB → event bus → embedding worker.
9. All tests pass (unit, integration, smoke). CI is green on the v0.1 release tag.
10. Every doc cross-link in the repo resolves (no dangling references).

## 4. Work breakdown

Eight work-streams. Each can land independently in PRs once the bootstrap (§4.1) is in.

### 4.1 Bootstrap (blocker for everything else)

- `pyproject.toml`, `src/openspine/` layout, `docker-compose.yml`, `Makefile`, `.env.example`, `.editorconfig`.
- FastAPI app skeleton with config, DB connection, OpenAPI tags per module.
- Alembic configured with one initial empty migration.
- Pre-commit hooks (ruff, mypy, conventional-commit lint, DCO check).
- GitHub Actions CI: lint → typecheck → test → build container.
- Owner: orchestrator. Council: not needed (uncontroversial scaffolding).

### 4.2 Identity core

- `id_*` migrations (tenant, principal, credential, session, token, audit_event, federated_identity stub).
- Password+TOTP auth + API token + agent token issuance.
- Principal context middleware sets request-scoped `tenant_id`, `principal_id`, `effective_roles`.
- RLS policies on every business table referencing `current_setting('openspine.tenant_id')`.
- Owner: `identity-expert` synthesises with `solution-architect`.
- Council: include `ai-agent-architect` for agent-token shape.

### 4.3 RBAC + auth-object engine

- `id_role_*`, `id_auth_object_*`, `id_permission`, `id_sod_rule`, `id_auth_decision_log`.
- `@requires_auth(domain, action, **qualifiers)` decorator.
- SoD baseline matrix loaded as system data on first start.
- Decision log writes async-batched.
- Owner: `identity-expert`.
- Council: every module expert reviews their domain's auth objects.

### 4.4 Master Data core

- `md_*` migrations for the 1.3 list.
- `MasterDataService` with create/read/update/list per entity.
- Number-range service. FX-rate upload endpoint.
- Posting-period open/close endpoints.
- Owner: `md-expert`.
- Council: `identity-expert` validates auth-object scoping for org units.

### 4.5 Event bus + embedding pipeline

- Redis Streams producer (`openspine.core.events`).
- Embedding worker (separate container) consuming `master_data.*` streams.
- Qwen3-Embedding via Ollama; per-tenant Qdrant collection naming.
- Reconciliation job + admin endpoint to trigger it.
- Owner: `solution-architect` (cross-cutting infrastructure).
- Council: `ai-agent-architect` for embedding-payload design (which fields, how plugin custom fields opt in via `visible_in: ["semantic_index"]`).

### 4.6 Plugin host

- Entry-point discovery, `plugin.yaml` parser with semver compatibility check.
- Hook registry + dispatcher (sync inline for `pre_*`, async via Redis for `post_*`).
- `extend_entity(entity, FieldDef)` — generates migration + UI/API/embedding wiring.
- Plugin-registered auth-object support.
- Example plugin in `examples/openspine-plugin-example/` exercising all four extension mechanisms.
- Owner: `plugin-architect`.
- Council: every module expert reviews the hook contract for their domain.
- **Blocker**: the hook-naming convention drift (`fi_document.*` vs `entity.action`) must be resolved by the council before the hook registry hardens. This is the single highest-stakes API decision in v0.1.

### 4.7 Agent surface

- Agent-token issuance with provenance + reasoning fields.
- `_meta` block on every response.
- Structured-error format enforced via FastAPI exception handlers.
- `id_agent_decision_trace` table + write API.
- Hybrid `/md/search` endpoint.
- Owner: `ai-agent-architect`.
- Council: `identity-expert` for token mechanics; `md-expert` for search semantics.

### 4.8 Observability

- OpenTelemetry instrumentation (auto for FastAPI, manual for event bus).
- Structured JSON logging (one logger config across processes).
- Prometheus metrics module exposing the metrics in §1.7.
- Owner: `solution-architect`.
- Council: not needed.

## 5. Sequencing

Sequential with parallelism inside each band:

1. **§4.1 Bootstrap** — must land first.
2. **§4.2 Identity core + §4.8 Observability** in parallel — both are infrastructure other streams need.
3. **§4.3 RBAC + §4.5 Event bus** in parallel — different surfaces, both needed before §4.4 hardens.
4. **§4.4 Master Data + §4.6 Plugin host** in parallel — biggest streams, run together once the bus and identity layers are in.
5. **§4.7 Agent surface** — last, because it depends on identity, master data, and event bus all being live.

A strict critical path is §4.1 → §4.2 → §4.3 → §4.4 → §4.7. Anything off this path can slip without slipping the milestone.

## 6. Risks

| Risk | Mitigation |
|------|------------|
| **Hook-naming convention drift hardens before resolution.** Once plugins exist with hook names baked in, renaming is a deprecation cycle. | Resolve in council before §4.6 starts. ADR-worthy. |
| **AGPL + private-plugin distribution legal nuance unaddressed.** Adopters block on this. | ADR `0004-agpl-license.md` lands with v0.1. |
| **RLS performance on hot paths.** Postgres RLS adds query-planner overhead; queries that touch many rows can regress. | Benchmark in §4.2; if regressive, fall back to service-layer-only filtering with RLS as belt-and-braces on lookup tables. ADR if so. |
| **Qwen3-Embedding via Ollama not fast enough for v0.1's reconciliation throughput.** | Fallback: smaller embedding model, or async batching in worker. Either is reversible in a later release. |
| **Multi-tenant Qdrant collection cardinality.** Many tenants → many collections → operational pain. | Decision matrix in `0002-qdrant-over-pgvector.md`: shared collection with tenant-keyed payload filtering vs collection-per-tenant. Probably collection-per-tenant for hard isolation, with reasonable cap; revisit if 100+ tenants is a real near-term target. |
| **Scope creep into v0.2 features.** Anyone touching FI is doing v0.2. | This doc's §2 is the contract. PRs adding `fin_*` get rejected with a "v0.2" label. |

## 7. Open questions

To be resolved before or during v0.1:

1. **Hook naming canonical form** — `entity.action` or `module.entity.action`? `plugin-architect` to mediate. (See §6.)
2. **Database-per-tenant vs shared schema with RLS** — `tenancy.md` open Q1. v0.1 default is shared+RLS; database-per-tenant flagged as a future deployment variant if regulated workloads demand.
3. **Qdrant collection topology** — see §6.
4. **Number-range scoping** — `md-master-data.md` open Q1. Centralised service per tenant with per-Company-Code scopes is the v0.1 plan; revisit during pilot.
5. **FX rate `mid` reference** — `permissions.md:73` says "mid-rate" but `md_exchange_rate_type` defines `M`/`B`/`G`. Resolve before v0.1 by either adding a `mid` rate type or changing the spec to use `M`. Belongs in `md-master-data.md` revision alongside §4.4.
6. **Audit-log topology** — `id_audit_event` (auth) vs `id_auth_decision_log` (authorisation) vs `id_agent_decision_trace` (agent reasoning). Document the join model in `docs/identity/README.md` before v0.1 ships.

## 8. Acceptance review

A v0.1 release-candidate goes live for review when §3 passes. The reviewer panel:

- A real ERP consultant (functional) walks through the master-data lifecycle and signs off that nothing essential is missing for v0.2 to start on top.
- A real ERP consultant (technical) writes a non-trivial plugin against the hook surface.
- A security reviewer audits identity, RLS, and audit-log completeness.
- An agent-design reviewer validates that the API truly is agent-consumable.

If any of those four says "no", the release candidate doesn't ship. v0.1 is the moment OpenSpine commits to its public contracts; getting it right matters more than getting it fast.

---

## 9. After v0.1

The path from v0.1 to v0.2 is then well-defined:

- v0.2 adds `fin_*` posting, the GL/AP/AR services, period management, and the React frontend.
- v0.2.x folds in the CO master tables and simple allocations.
- The hook naming, plugin contract, agent surface, and dual-write pipeline are **frozen** at v0.1 — they evolve only through the deprecation cycle.

That stability is what makes the rest of the roadmap predictable. Spending the time on v0.1 to lock these in is the whole point.
